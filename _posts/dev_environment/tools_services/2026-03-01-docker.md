## 사용 이유
 
### 멀티 플랫폼 문제
    - 윈도우, 맥, 리눅스(종류 진짜 많음) 등 전부 환경이 다르고 서로 호환되지 않는다.
    - 때문에 플랫폼별로 프로그램 설치 및 환경 설정을 해야하는 과정이 제각각 많이 달라진다. (당장에 윈도우는 zip으로 풀거나 exe 쓰면 되는데, 리눅스는 명령어를 써야 한다...)
<br>

### 셋팅 문제
- NestJS를 쓴다고 치자. 그러면 nodejs 설치, yarn을 쓰면 yarn도 설치, 다 했으면 nestjs CLI 설치, DB 쓸거면 DB도 설치해야 한다.
- 사실 위 과정이 진짜 문제다. 저 과정을 플랫폼이 달라질때마다 계속해서 거쳐야 하고 플랫폼마다 세부 사항이 달라지기까지 한다. 처음 하는 사람들이라면 애초에 익숙치가 않으니 여기서 막힐거고 익숙한 사람은 이 과정이 상당히 귀찮게 느껴질 것이다. (당장에 내가 우분투에서 고작 파이썬 다시 설치할 때 엄청 귀찮았던걸 생각하면...)

대표적인건 아마 위에 2가지로 충분하고도 남을 것이다.
<br><br>



## docker file
 - 그래서 만들어진 개념이 **docker file**이다.
 - 이 파일 내용에는 이런저런 설치의 절차를 작성하는데, 쉽게 말하면 <ins>무조건 이 순서대로 따라가면 개발자가 프로젝트를 실행할 수 있게 되는 정의</ins>를 할수 있는 파일이라고 보면 된다.
 - 컨테이너라는 개념이 여기서 나오게 된다. <ins>특정 프로그램을 실행을 할 때 필요한 환경과 패키지들, 명령어들을 하나의 컨테이너. 즉, **도커 파일**이라는 하나의 컨테이너로 묶어 어느 환경에서든 사용한다</ins> 라고 알면 된다.
 - 이 파일을 도커에 넣으면 알아서 절차에 따라 설치하고 설정하고 실행되게 된다. 어느 환경에서든 파일 하나로 사용이 가능하게 되는 것이다.

### VM
Hypervisor를 통해 각각 프로그램을 실행을 할 환경에서 실행할 OS를 추가로 설치해야 한다. 그래서 Guest OS가 추가로 하나 올라가져 있는걸 볼 수 있다.

### Docker
VM과 다르게 Host OS 커널이랑 소통을 하기 때문에 Guest OS라던가 필요가 없게된다. OS 설치를 추가로 안해도 되니 개발 효율이 빨라지게 된다.
<br><br>



## 기초 예제
NestJS 기준으로 설명한다. (강의 때문인것도 있다.)
 - CLI 열고 docker --version 되는지 확인
 - 컨테이너 설정을 위해 Docker Hub에서 필요한 컨테이너 검색 후 이름/버전을 기억
 - 사용할 프로젝트의 최상위 단에 docker-compose.yaml 파일 추가
 - 파일 내용은 아래와 같다.
```yml
service:
  postgres:
        image: postgres:15 # 이름:버전
        restart: always # 실행 시 매번 재시작
        volumes: # 해당 도커 이미지 안에서 실행됐던 또는 생성됐던 파일들은 도커 종료 시 다 날아간다. 이걸 로컬에 저장시켜 데이터를 지키기 위한 방식
            - ./postgres-data:/var/lib/postgresql/data # hostOS의 postgres-data 폴더 안에 이미지에 해당하는 경로를 저장하겠다는 이야기 (hostOS 경로:이미지 경로)
        ports: # 이미지 속 포트 == hostOS 포트
            - "5432:5432" # hostOS 포트:이미지 속 포트
  environment: # DB 관련 정보
	    POSTGRES_USER: postgres
	    POSTGRES_PASSWORD: postgres
	    POSTGRES_DB: postgres
```
{: file='docker-compose.yaml' }
 - CLI로 가서 docker-compose up 실행 (up 실행, down 종료)
 - 설정한 폴더에 postgres 관련 내용들이 잔뜩 생겼으면 성공이다.
